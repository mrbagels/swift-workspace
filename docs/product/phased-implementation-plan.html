Phased Implementation Plan

A staged path from clean scaffold to professional reusable workspace product, using the old Mac Shell as prototype reference while building the new source of truth in swift-workspace.

Open Markdown Download Markdown

Phase 0: Fresh Root And Guardrails

Complete for the initial scaffold: new root, package, XcodeGen project, docs lifecycle, first products, demo apps, and verification scripts.

Phase 1: Stabilize WorkspaceCore

Complete for the first stable engine vocabulary: command roles and grouping, route metadata patches, URL route parsing, Codable storage-safe models, and expanded core tests are in place. Continue API review before public tags.

Phase 2: Make WorkspaceTCA Boring

Complete for the current shared reducer surface: reducer tests cover palette lifecycle, recents, policy, route-open behavior, scene delegates, restoration, and metadata reconciliation.

Phase 3: Port Proven Mac Shell Behavior

Complete for the first professional Mac renderer baseline: Mac shell configuration, native split rendering, command palette UI, command reference UI, command-menu entry points, Mac-specific restoration, accessibility identifiers, focus hardening, and visual-state fixtures are in place.

Phase 4: Build The iOS And iPadOS Renderer

Complete for the first adaptive iOS and iPadOS renderer baseline: iOS shell configuration, restoration, command-search sheet, badge preferences, scene-aware route actions, demo adoption, accessibility identifiers, focus hardening, and compact/split visual-state fixtures are in place.

Phase 5: Persistence And iCloud Primary Storage

Complete for persistence helpers and CloudKit contracts: file-backed restoration storage, CloudKit record names, zones, conflict policy, restoration envelopes, route metadata envelopes, adapter contracts, tests, and adoption docs are in place. Live CloudKit sync remains app-owned.

Phase 6: Companion Server

Decision-gated: server scope, anti-scope, initial API areas, and implementation gates are documented. Do not add WorkspaceServerClient until a consuming app proves the first server-backed workflow.

Phase 7: Distribution And Adoption

In progress: a compiled custom-renderer client example, adoption docs for every current adoption path, API review checklist, release checklist, and verification coverage for the example package are in place. Public release work still needs versioning and repository decisions.

Phase 8: Professional Polish

In progress: automation anchors and visual-state fixtures are in place. Manual design, interaction, and demo review remain before public release.

Current Remaining Work

Autonomous work: DocC catalogs, CI workflow once the runner is known, and fixture expansion as public behavior grows. Decision-gated work: first server workflow, package version and public repository URL, starter app shape, and hands-on Mac/iOS demo review.
